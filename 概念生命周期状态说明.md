当前实际实现采用：

> 使用过的概念继续保持 `status=ready`，通过 `usage-log`、`last_used` 和 `use_count` 表示“已使用”。

如果使用后将 `status` 改为 `used`，`ai-news-factory` 的 eligible 筛选会排除该概念，使其在冷却结束后也无法再次参与自动选题。因此，当前 `used` 更适合理解为一次使用事件或运营状态，而不是持久化的内容状态。

### 进入条件

一个概念满足以下情况时，可以进入 `candidate`：

1. 在日报、周报、月报或视频脚本中真实出现。
2. 在 `extracts/term-frequency.*` 中有语料证据，或者出现在日报“概念候选”附录。
3. 不懂该概念可能影响普通观众理解新闻。
4. 有机会在约 15 秒内讲清一个核心点。
5. 不是已有概念的简单别名或重复卡片。
6. 不是公司名、模型型号、单次事件名称或权益黑话。

最小字段示例：

```json
{
  "id": "model_routing",
  "name": "模型路由/调度",
  "status": "candidate",
  "news_keywords": [
    "模型路由",
    "模型调度"
  ]
}
```

### 允许操作

- 补充 `aliases` 和 `news_keywords`。
- 补充语料证据 `corpus` 和 `sources`。
- 调整建议分类、难度和 tier。
- 判断它应成为独立概念，还是已有概念的新别名或新角度。
- 不值得继续维护时删除候选，但应确认没有下游引用其 `id`。
- 确认值得生产后晋升为 `draft`。

### 禁止操作

- 禁止被 `linuxdo-daily` 或 `ai-news-factory` 当成正式锚点。
- 禁止从 `candidate` 直接跳到 `ready`，绕过 narrator 和审核。
- 禁止人工随手写一段口播后标记为成品。
- 禁止因为出现频率高就默认技术事实正确。
- 禁止把产品名、型号或社区黑话包装成技术概念。
- 禁止建立与现有 `id`、`name` 或 `aliases` 重复的概念卡。
- 禁止伪造 `script_meta.authored_at` 或审核记录。

## 二、`draft`：已立项、待生产或待审核

### 当前数量

当前为 0。

这意味着当前概念库呈现为两端：

```text
27 个 candidate
20 个 ready
```

中间没有正在生产或等待审核的条目。

### 进入条件

`candidate` 满足以下条件后，可以晋升为 `draft`：

1. 已确认它是值得长期维护的技术概念。
2. 已排除与现有概念重复。
3. 已确定稳定 `id` 和展示名称。
4. 已补充基本元数据：
   - `category`
   - `tier`
   - `difficulty`
   - `aliases`
   - `news_keywords`
   - `angles.available`
5. 有语料证据或可靠来源。
6. 准备调用 narrator 生成口播，或者 narrator 已生成但尚未审核。

调用 narrator 前的典型状态：

```json
{
  "status": "draft",
  "one_liner": "",
  "analogy": "",
  "script_15s": "",
  "script_meta": {
    "authored_by": "ai-concept-narrator",
    "authored_at": null,
    "reviewed": false,
    "angle": null
  }
}
```

调用 narrator 写入之后：

```json
{
  "status": "draft",
  "one_liner": "……",
  "analogy": "……",
  "script_15s": "……",
  "script_meta": {
    "authored_by": "ai-concept-narrator",
    "authored_at": "2026-07-13",
    "reviewed": false,
    "angle": "基础定义",
    "confidence": "high",
    "forbidden_check": "pass"
  }
}
```

即使台词已经生成，只要没有经过明确审核，仍然必须停留在 `draft`。

### 允许操作

- 调用 `ai-concept-narrator` 生成口播。
- 修改或补充 `news_keywords`、`aliases`、`sources`。
- 指定一个未使用的讲解角度。
- 核实 narrator 返回的事实。
- 检查字数、TTS 可读性和违禁词。
- 审核不通过时重新调用 narrator。
- 审核通过后改为 `ready`。

### 禁止操作

- `script_meta.reviewed=false` 时禁止标记为 `ready`。
- `confidence=low` 时禁止标记为 `ready`。
- narrator 列出 `facts_to_verify` 且尚未核实时禁止上线。
- 禁止让主会话自行写完整口播替代 narrator。
- 禁止从归档草稿、旧白说库或其他模型输出直接复制成品。
- 禁止在审核前写入 `usage-log.json`。
- 禁止更新 `last_used` 和 `use_count`，因为它还未实际使用。
- 禁止为了赶当期视频而省略事实核验。

### 进入条件

只有同时通过以下质量门，`draft` 才能进入 `ready`：

```text
status == ready
script_15s 非空
script_meta.authored_by == ai-concept-narrator
script_meta.reviewed == true
script_meta.confidence != low
```

同时应满足：

1. `one_liner` 非空。
2. `analogy` 非空。
3. `script_15s` 目标 60–80 字，约 100 字以内。
4. 口播包含概念点题、白话定义、生活化比喻、价值或新闻关系。
5. 无违禁词。
6. 技术事实已经核实。
7. 有明确的 `script_meta.angle`。
8. 尽量有 `corpus` 或 `sources` 作为追溯依据。

- 被 `linuxdo-daily` 引用 `one_liner`。
- 被 `ai-news-factory` 选为专业锚点。

- 周报、月报优先使用 `script_60s`，为空时使用 15 秒兜底。
- 根据 `news_keywords`、`aliases` 和 `name` 匹配新闻。
- 根据 `last_used` 和 `reuse_gap_days` 判断冷却期。
- 冷却结束后再次使用。
- 换角度时重新调用 narrator。
- 发现内容过时后改为 `stale`。

### 禁止操作

- 禁止下游修改库内口播正文。
- 禁止 `ai-news-factory` 自行重写完整专业解释。
- 禁止未经用户确认就写 `usage-log`。
- 禁止为了命中新闻临时篡改 `news_keywords`。
- 禁止绕过冷却期并删除 `last_used`。
- 禁止重复使用同一角度但不记日志。
- 禁止直接修改 `authored_by` 冒充 narrator。
- 禁止把 `reviewed=false` 自动改为 `true`。
- 禁止继续使用已经确认过时的台词，应转为 `stale`。

## 四、`used`：实际已使用

### 当前数量

`concepts.json` 中：

```text
status=used：0 个
```

但从 `usage-log.json` 看，实际使用过 3 个概念：

```text
agent
model_dumbing
token
```

### 当前实际设计

当前应把“已使用”理解为以下组合：

```text
status = ready
+ usage-log 中存在记录
+ last_used 非空
+ use_count > 0
+ angles.used 包含使用角度
```

而不是强制：

```text
status = used
```

原因是 `ai-news-factory` 的 eligible 条件要求 `status == ready`。如果使用后改成 `used`，它将无法在冷却期结束后重新参与自动选题。

推荐的实际流程：

```text
ready（可用）
  ↓ 实际使用
仍保持 ready
  + append usage-log
  + 更新 last_used
  + use_count += 1
  + angles.used 追加角度
```

### 进入条件

一次使用只有同时满足以下条件，才算真正进入“已使用”运营状态：

1. 概念原本 eligible。
2. 已进入最终视频或报告脚本。
3. 完整脚本已经用户确认。
4. 确实准备进入后续制作流程。
5. 已向 `usage-log.json` 追加记录。
6. 已更新 `last_used`。
7. 已递增 `use_count`。
8. 已记录本次使用角度。

### 允许操作

- 保持 `status=ready`，等待冷却期后再次使用。
- 查看历史使用角度。
- 根据下次新闻选择新角度。
- 根据用户反馈调整 `angles.available`。
- 使用新角度时重新调用 narrator。
- 记录强制突破冷却期的原因到 `notes`。

### 禁止操作

- 禁止覆盖旧 `usage-log` 记录。
- 禁止删除旧角度来伪装成首次使用。
- 禁止只更新 `use_count` 而不写日志。
- 禁止只写日志而不更新 `last_used`。
- 禁止重复追加同一个 `angles.used` 值。
- 禁止用户尚未确认脚本时提前记账。
- 禁止视频取消制作后仍保留虚假的上线记录。
- 禁止直接改成 `used` 导致概念退出 eligible，除非同步修改整个下游筛选规则。

### 当前数据一致性风险

当前日志记录次数为：

```text
agent：usage-log 1 条
model_dumbing：usage-log 1 条
token：usage-log 1 条
```

但 `agent.use_count` 当前为 2，说明 Agent 的统计和现有 usage log 不完全一致。

可能原因包括：

- 曾使用过一次但旧日志没有补录。
- `use_count` 被手动增加过。
- 某次使用记录被遗漏。
- 数据初始化时保留了历史次数。

这是需要确认的数据一致性问题，不应静默把 `use_count` 改回 1。

## 五、`stale`：内容过时、暂停使用

### 当前数量

当前为 0。

### 进入条件

一个 ready 概念出现以下情况时，应标为 `stale`：

1. 技术定义已经发生变化。
2. 台词包含已经过时的数据或案例。
3. 产品行为变化导致原有解释不再准确。
4. 原来的类比会产生明显误导。
5. 新事实证明旧口播存在错误。
6. `sources` 失效，且关键结论无法重新验证。
7. 旧口播使用了不再合适的措辞。
8. 概念边界与相邻概念发生冲突，需要重构。
9. `confidence` 应降为 `low`。
10. 用户或观众反馈指出关键理解错误。

稳定基础概念通常不容易整体过时，但其中的数字、计费示例和产品案例可能过时。

### 允许操作

- 暂停下游使用。
- 补充 `facts_to_verify`。
- 重新读取语料和可靠来源。
- 重新选择讲解角度。
- 调用 narrator 重写口播。
- 重写后先回到 `draft`。
- 明确审核通过后再回到 `ready`。

推荐流程：

```text
ready
  ↓ 发现错误或过时
stale
  ↓ 核实事实、确定新角度
draft
  ↓ narrator 重写
draft + reviewed=false
  ↓ 明确审核
ready
```

### 禁止操作

- 禁止 `stale` 概念继续进入日报或视频锚点。
- 禁止只修改一句话就直接恢复 `ready`。
- 禁止保留旧 `reviewed=true` 来表示新台词也已审核。
- 禁止重写台词后不更新 `authored_at`。
- 禁止用旧新闻案例证明当前仍然成立。
- 禁止删除历史使用记录。
- 禁止删除旧使用角度。
- 禁止因为概念曾经表现好就忽略事实过时问题。

## 推荐的实际状态机

结合当前下游筛选规则，建议把生命周期明确为：

```text
candidate
    ↓ 确认值得维护
draft
    ↓ narrator 生成 + 明确审核
ready
    ↓ 用户确认并实际使用
ready + usage-log + last_used + use_count
    ↓ 过冷却期
ready，可再次选择
    ↓ 内容过时
stale
    ↓ 核实并准备重写
draft
    ↓ narrator + 审核
ready
```

更准确的表达不是：

```text
candidate → draft → ready → used
```

而是：

```text
candidate → draft → ready
                     ├─ 使用事件 → 仍为 ready，并写 usage-log
                     └─ 内容过时 → stale → draft → ready
```

## 各状态的下游使用权限

| 状态 | `linuxdo-daily` 可引用 | `ai-news-factory` 可上片 | 可写 usage log |
|---|---:|---:|---:|
| `candidate` | ❌ | ❌ | ❌ |
| `draft` | ❌ | ❌ | ❌ |
| `ready` | ✅ `one_liner` | ✅ 完整口播 | 用户确认使用后 ✅ |
| `used` | 当前不建议作为持久 status | 当前 eligible 会排除 | 已有历史日志 |
| `stale` | ❌ | ❌ | ❌ |

## 最终建议

- `status` 负责表示内容是否可用：`candidate`、`draft`、`ready`、`stale`。
- `usage-log`、`last_used`、`use_count` 负责表示概念是否使用过。
- 不再将 `used` 作为持久 `status`，除非同步修改 `ai-news-factory` 的 eligible 条件。
- 定期检查 `use_count` 与 `usage-log` 的一致性；发现差异时先确认历史，不要静默修正。
