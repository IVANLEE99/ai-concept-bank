# AI 概念库建立与维护方案

> 概念库不是 AI 百科全书，而是服务于日报选题和 15 秒专业解释的内容工具。

---

## 一、核心策略

建立概念库时，采用两条线并行：

```text
日报持续提取 + 基础概念地图补漏
```

- **日报持续提取**：保证概念与近期新闻、观众兴趣相关。
- **基础概念地图**：避免选题完全被热点牵着走，长期保持知识覆盖面。
- **难度分层**：控制内容节奏，不要求每天机械地从浅到深讲。
- **使用记录**：避免短期重复，并支持同一概念从不同角度复用。

每个正式概念最终都应该能够回答三个问题：

```text
什么新闻出现时讲？
15 秒具体怎么讲？
最近是否已经讲过？
```

## 二、概念库字段

初期使用 Markdown 表格维护即可，不必马上引入数据库或复杂工具。

| 字段 | 作用 |
|---|---|
| 概念 | 统一使用的标准名称 |
| 别名 | 新闻中可能出现的中英文名称、缩写或其他叫法 |
| 分类 | 模型基础、训练、Agent、安全、评测、商业等 |
| 难度 | 入门、进阶、专业 |
| 一句话解释 | 判断是否能在 15 秒内讲清 |
| 新闻触发词 | 用于从日报中匹配概念 |
| 适合场景 | 哪类新闻出现时适合讲 |
| 可讲角度 | 同一概念可以拆出的不同选题 |
| 已讲角度 | 防止重复讲相同内容 |
| 状态 | 候选、待核实、可使用、已排期、已使用、需更新 |
| 上次使用 | 控制复用间隔 |
| 使用次数 | 观察概念分布是否失衡 |
| 关联概念 | 为后续选题提供延伸方向 |
| 信息来源 | 记录权威资料，方便核实和更新 |

示例：

| 概念 | 别名 | 分类 | 难度 | 一句话解释 | 新闻触发词 | 状态 | 上次使用 |
|---|---|---|---|---|---|---|---|
| MoE | 混合专家模型、Mixture of Experts | 模型架构 | 进阶 | 每次只激活部分专家模块，在扩大模型能力的同时控制计算成本 | MoE、专家模型、激活参数 | 可使用 | 2026-07-01 |
| 上下文窗口 | Context Window | 模型基础 | 入门 | 模型一次能够读取和处理的信息量 | 128K、1M Token、长上下文 | 可使用 | - |
| API Key | API 密钥 | 安全 | 入门 | 调用 AI 服务时用于验证身份和计费的凭证 | 密钥泄露、额度被刷、GitHub 泄露 | 可使用 | - |

## 三、概念来源

### 1. 从日报中持续提取

每天整理日报时，顺手标记：

- 新闻明确提到的技术名词。
- 普通观众可能看不懂的缩写。
- 新闻结论背后的技术原因。
- 评论区可能继续追问的问题。
- 同一周反复出现的关键词。

示例：

```text
新闻：某模型支持 100 万 Token 上下文

显性概念：Token、上下文窗口
隐性概念：注意力机制、KV Cache、上下文压缩
当期优先解释：上下文窗口
其他概念：进入候选区
```

不要把所有出现过的词直接放进正式库。新词先进入候选区，确认有内容价值后再完善。

### 2. 建立基础概念地图

日报提取容易受热点影响，因此需要通过固定分类补漏。

#### 基础认知

- Token
- 参数
- 上下文窗口
- 模型幻觉
- 多模态

#### 模型训练

- 预训练
- 微调
- 蒸馏
- RLHF
- 合成数据

#### 模型架构

- Transformer
- Attention
- MoE
- Tokenizer
- KV Cache

#### 模型应用

- RAG
- Embedding
- Function Calling
- Agent
- MCP

#### 推理与部署

- 推理模型
- 模型量化
- 并发
- 延迟
- 显存
- API

#### 安全

- API Key
- 提示词注入
- 越权操作
- 数据泄露
- 沙箱

#### 评测

- Benchmark
- 数据污染
- 刷榜
- HumanEval
- 真实任务评测

#### 开源与商业

- 开放权重
- 开源协议
- API 商业化
- 私有化部署
- 云服务

基础概念地图负责保证覆盖面，日报负责决定当期实际讲什么。

## 四、正式收录标准

一个概念至少满足以下两项，才进入正式概念库：

- 最近 AI 新闻中频繁出现。
- 不理解它会影响观众理解新闻。
- 可以在 15 秒内讲清一个核心点。
- 能对应具体案例或生活类比。
- 对普通 AI 用户或开发者有实际价值。
- 能自然关联后续概念。

以下内容暂不收录：

- 只出现一次的产品内部术语。
- 15 秒内无法定义清楚的学术细节。
- 与目标观众无关的底层数学概念。
- 只有定义、没有实际意义的名词。
- 含义不明确或尚未证实的营销词。

## 五、概念生命周期

为每个概念设置明确状态：

```text
候选 → 待核实 → 可使用 → 已排期 → 已使用 → 待复用 / 需更新
```

### 候选

刚从日报、评论区或行业动态中发现，尚未判断是否值得讲。

### 待核实

已经确定有选题价值，但技术定义、数据或案例仍需核实。

### 可使用

已经具备准确解释、触发场景和脚本方向，可以随时选用。

### 已排期

已经与某期日报匹配，防止同一个概念被重复安排。

### 已使用

记录发布日期、关联新闻、讲解角度和内容表现。

### 待复用

已经达到复用间隔，并且还有新的讲解角度。

### 需更新

定义、数据、案例或产品信息可能已经过时，需要重新核实。

## 六、每期选题评分

当多个概念都适合时，可以按五项指标评分，每项 0～2 分。

| 维度 | 判断问题 |
|---|---|
| 新闻关联度 | 是否直接出现在当天日报中？ |
| 理解价值 | 不解释是否会影响观众理解？ |
| 热度 | 最近是否频繁出现？ |
| 可讲性 | 15 秒内是否能讲清？ |
| 新鲜度 | 最近是否没有讲过？ |

计算方式：

```text
总分 = 新闻关联度 + 理解价值 + 热度 + 可讲性 + 新鲜度
```

建议规则：

```text
8～10 分：优先使用
6～7 分：可以使用
4～5 分：保留在候选区
0～3 分：暂不使用
```

评分用于辅助决策，不必把内容生产变成复杂的数据系统。

## 七、防止重复

设置三层去重规则：

1. 完全相同的概念至少间隔 14～30 天。
2. 同一分类不要连续超过 2 期。
3. 已讲概念可以换角度复用，但不能重复相同定义。

例如，MoE 可以拆成多个角度：

```text
第一次：什么是 MoE
第二次：为什么 MoE 能降低推理成本
第三次：总参数和激活参数有什么区别
第四次：MoE 为什么可能出现专家负载不均
```

概念卡片中增加角度记录：

```text
已讲角度：
- 基础定义
- 成本优势

待讲角度：
- 激活参数
- 路由机制
- 潜在缺点
```

## 八、维护频率

### 每天：收集

整理日报时，把新概念放进候选区。

只记录名称、关联新闻和来源，不要求当天写完整解释。建议控制在 2～3 分钟。

### 每周：清理

每周集中处理一次：

- 合并重复概念和中英文别名。
- 删除没有持续价值的营销词。
- 补全高频概念的一句话解释。
- 核实技术定义和新闻案例。
- 更新概念使用记录。
- 为下一周准备 3～5 个备用概念。

### 每月：复盘

根据内容数据调整概念库：

- 哪类概念完播率更高。
- 哪类概念获得更多收藏和评论。
- 哪些解释被认为太难。
- 哪些概念已经重复过多。
- 哪些分类长期没有覆盖。
- 哪些旧解释需要更新。

## 九、内容数据记录

每个已经使用的概念可以记录：

```text
发布日期：
关联新闻：
讲解角度：
视频播放量：
专业解释开始处留存：
专业解释结束处留存：
完播率：
收藏率：
评论关键词：
观众是否表示没听懂：
是否值得换角度复用：
```

复盘时不要只看播放量，重点关注：

- 专业解释开始后是否出现明显掉线。
- 该段是否带来收藏、转发和追问。
- 哪类概念更容易获得正向反馈。
- 哪种解释结构更容易被观众理解。

如果专业解释段经常导致掉线，优先检查：

- 定义是否过于抽象。
- 15 秒内是否出现太多专业名词。
- 概念是否与当天新闻脱节。
- 句子是否太长。
- 是否同时解释了多个概念。

## 十、推荐文件结构

初期维护两个文件即可：

```text
social_media/
├── AI概念库.md
└── AI概念使用记录.md
```

### AI概念库.md

负责记录：

- 概念定义
- 分类和难度
- 新闻触发词
- 适用场景
- 可讲角度
- 15 秒脚本
- 信息来源

### AI概念使用记录.md

负责记录：

- 使用日期
- 对应日报
- 使用角度
- 内容表现
- 观众反馈
- 下次可复用时间

## 十一、最小执行流程

```text
1. 每天从日报提取候选概念
2. 每周筛选并补全高价值概念
3. 每期按关联度和可讲性选一个
4. 使用后记录日期、角度和数据
5. 每月根据观众反馈清理概念库
```

最终原则：

> 少收录、持续更新、每个概念都能直接转化成一期 15 秒内容。
